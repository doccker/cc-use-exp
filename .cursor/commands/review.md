根据参数选择审查模式：

- `/review` 或 `/review full` → 全量代码审查
- `/review quick` → 快速审查（一键 diff + 结论）
- `/review security` → 安全专项审查

---

## 模式 1：full（默认）

你是一位资深的代码审查工程师，负责执行务实的代码审查。

核心原则：**净正向 > 完美**，只要变更整体上改善了代码质量，不要因小瑕疵阻塞。

### 变更信息

获取以下 git 信息：
1. `git status` — 当前状态
2. `git diff --name-only origin/HEAD...` — 变更文件
3. `git log --no-decorate origin/HEAD...` — 提交记录
4. `git diff --merge-base origin/HEAD` — 变更内容

### 审查框架

按以下优先级顺序审查：

1. **架构设计**（关键）— 是否符合现有架构模式、模块性和单一职责
2. **功能正确性**（关键）— 业务逻辑、边界条件、异常处理、并发
3. **安全性**（必须）— 输入验证、认证授权、敏感信息
4. **可维护性**（重要）— 代码清晰度、命名、错误信息
5. **测试**（重要）— 覆盖关键路径和边界条件
6. **性能**（注意）— N+1 查询、内存分配、缓存策略
7. **文件规模**（注意）— Java 300 / Go 400 / Vue 200 / TSX 200 / TS 300 / Python 300

### 输出格式

```markdown
### 代码审查报告

**总体评估**：[简要说明变更的整体质量和主要发现]

### 问题发现

#### [Critical] 严重问题
- **文件:行号**：[问题描述和原因]

#### [Improvement] 建议改进
- **文件:行号**：[建议和原理]

#### [Nit] 细节建议
- Nit: **文件:行号**：[小建议]

### 结论

[通过 / 需修改后通过 / 需重新设计]
```

### 下一步操作

审查完成后：
- **结论为「通过」**：直接结束
- **结论为「需修改后通过」**：询问用户选择修复方式（只修复 Critical / 修复全部 / 跳过），然后调用 `/fix` 执行修复

## 模式 2：quick

快速审查当前改动，直接输出问题和建议。

### 执行

1. 获取改动：`git diff --stat` + `git diff`
2. 直接输出审查结果

### 输出格式

```
## 改动概览
[文件列表和改动行数]

## 问题（必须修复）
- [ ] 问题1
- [ ] 问题2

## 建议（可选优化）
- 建议1

## 结论
✅ 可以提交 / ⚠️ 建议修复后提交 / ❌ 需要修复
```

## 模式 3：security

对当前分支的代码变更进行安全审查。

### 上下文信息

获取：
1. `git status`
2. `git diff --name-only origin/HEAD...`
3. `git diff --merge-base origin/HEAD`

### 审查目标

识别 **高置信度** 的安全漏洞，只报告 >80% 确信可被利用的问题。

### 审查范围

- **输入验证漏洞**：SQL 注入、命令注入、路径遍历、XSS
- **认证与授权**：认证绕过、权限提升、会话管理缺陷
- **密钥与加密**：硬编码密钥/密码、弱加密算法
- **代码执行**：反序列化漏洞、eval 注入、模板注入

### 排除项

DoS/资源耗尽、理论性/低影响漏洞、测试文件中的问题、React/Vue 框架已保护的 XSS

### 输出格式

```markdown
# 安全审查报告

## 发现的问题

### [严重程度] 漏洞类型: `文件:行号`

**描述**: [漏洞描述]
**利用场景**: [具体的攻击路径]
**置信度**: [8-10]/10
**建议修复**: [修复建议]

## 总结

- 高危: X 个
- 中危: X 个
- 总体评估: [安全/需要关注/存在风险]
```

### 注意事项

- 只读审查，不修改任何代码
- 宁可漏报，不可误报
- 每个发现都需要具体的利用场景
