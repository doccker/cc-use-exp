---

根据参数选择扫描模式：

- `/optimize` 或 `/optimize full` → 六维全量评估
- `/optimize ux` → 仅 UX 与操作费力度
- `/optimize perf` → 仅性能与中间件评估
- `/optimize code` → 仅代码质量、语法糖与冗余

---

## 第 0 步：范围选择

询问用户扫描范围：
1. 全量扫描（前端 + 后端）
2. 仅后端
3. 仅前端
4. 指定目录

全量扫描时先统计规模（文件数、行数），提示 token 消耗后确认继续。

---

## 第 1 步：项目画像

扫描项目基本信息：技术栈、框架、数据库、中间件、文件规模、大文件检测。

---

## 第 2 步：六维评估

| 模式 | 执行维度 |
|------|---------|
| `full` | UX + PERF + CODE + SUGAR + SEC + BEST |
| `ux` | UX |
| `perf` | PERF + BEST（中间件相关） |
| `code` | CODE + SUGAR |

### [UX] 用户体验与操作费力度
表单防重复提交、列表空状态、加载状态、错误提示、表单校验、批量操作、响应式布局

### [PERF] 性能与中间件评估
N+1 查询、全表查询、缺失索引、路由懒加载、大依赖检测；Redis/MinIO/ES/MQ 引入决策矩阵

### [CODE] 代码质量与组件抽取
重复组件/逻辑、上帝文件、死代码、硬编码；给出组件抽取建议

### [SUGAR] 语法糖与现代化写法
按 Go/Java/Vue/TS/Python 版本检测可用的新语法

### [SEC] 安全漏洞
SQL 注入、命令注入、路径穿越、XSS、硬编码密钥、CORS、不安全加密

### [BEST] 业界实践参考
依赖版本、框架最佳实践、项目结构、API 设计、错误处理、日志规范

---

## 第 3 步：输出报告

优先级排序：
```
[Critical]  — SEC 安全漏洞、PERF N+1/全表扫描、功能缺陷
[Important] — UX 体验问题、CODE 重复代码、PERF 中间件建议
[Nice]      — SUGAR 语法优化、BEST 实践建议
```

---

## 第 4 步：用户选择修复范围

1. 修复全部 Critical（推荐）
2. 修复 Critical + Important
3. 我来选择要修的项
4. 仅保留报告

---

## 第 5 步：逐项修复 + 逐项验证

- 一项一改，一项一验证，不批量修改
- SEC 优先、SUGAR 最后
- 修复前 `git stash` 备份，失败则回滚

---

## 第 6 步：最终验证 + 变更汇总

运行全量测试/构建，输出改前改后对比表和变更清单。
