根据参数选择设计模式：

- `/design doc [功能]` → 生成技术设计文档框架
- `/design checklist [功能]` → 生成设计质量检查清单
- `/design [功能]` → 默认生成文档框架（同 doc）

---

## 模式 1：doc（默认）

生成技术设计文档框架，包含以下章节：

1. **需求背景** — 业务背景和目标
2. **核心原则** — 3-5 条指导设计决策的原则
3. **术语定义** — 关键术语说明（如需要）
4. **数据库设计** — 新增/修改表、字段说明
5. **功能设计** — 功能模块、界面设计、处理逻辑
6. **接口设计** — API 定义、请求/响应格式、实现位置
7. **前端设计** — 组件结构、页面实现
8. **影响范围评估** — 代码改动范围、风险评估
9. **测试要点** — 功能测试清单、测试用例
10. **部署/迁移说明** — 数据库迁移、配置变更

根据实际需求填充内容，不需要的章节可省略。

## 模式 2：checklist

基于 S-Tier SaaS 设计标准（参考 Stripe、Airbnb、Linear），进行设计质量检查。

### 检查清单

#### 一、核心设计原则

- [ ] **用户优先**：界面是否以用户任务为中心设计
- [ ] **简洁清晰**：信息层次是否清晰，无冗余元素
- [ ] **一致性**：颜色、字体、组件是否与现有系统一致
- [ ] **可访问性**：对比度是否达标（WCAG AA 4.5:1）

#### 二、视觉设计

**颜色**
- [ ] 主色调使用是否克制（不超过 3 种主色）
- [ ] 语义色是否正确（成功=绿、错误=红、警告=黄、信息=蓝）

**字体**
- [ ] 标题层级是否清晰（H1 > H2 > H3 > Body）
- [ ] 正文字号是否舒适（14-16px）
- [ ] 行高是否合适（1.5-1.7）

**间距**
- [ ] 是否使用 8px 基准网格
- [ ] 元素间距是否一致

#### 三、交互设计

- [ ] 按钮状态是否完整（默认、悬停、激活、禁用）
- [ ] 表单验证反馈是否及时
- [ ] 加载状态是否有提示
- [ ] 空状态是否有引导
- [ ] 错误状态是否有恢复建议

#### 四、响应式

- [ ] 桌面端（1440px）布局是否正常
- [ ] 平板端（768px）是否适配
- [ ] 移动端（375px）是否可用

#### 五、性能

- [ ] 图片是否优化（WebP、懒加载）
- [ ] 动画是否流畅（60fps）
- [ ] 首屏加载是否快速

### 输出格式

```markdown
### 设计检查报告：[功能名称]

**总体评分**：X/5

#### ✅ 通过项
- [列出通过的检查项]

#### ⚠️ 建议改进
- [问题描述] → [改进建议]

#### ❌ 必须修复
- [问题描述] → [修复方案]
```
