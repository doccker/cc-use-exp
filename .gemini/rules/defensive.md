# 防御性编码与质量规则

> 预防常见 AI 编码陷阱，基于工程最佳实践。

---

## 🚫 严格禁止

### 1. 测试篡改（最严重）
- ❌ 修改测试断言来适配错误实现
- ❌ 删除或注释掉失败的测试
- ✅ 测试失败时，必须修复实现代码，保持测试集的完整性

### 2. 过度工程化
- 简单 if/else 优于复杂抽象
- 不要引入未明确要求的模式（如“工厂”、“策略”），除非项目已存在大规模同类用法

### 3. 盲目修改代码
修改前必须：
- 搜索类似功能，了解项目现有实现模式
- 检查修改点的所有上游调用，评估回归风险

### 3.5 重构代码（高风险）
**重构前必须**：
- 完整读取原始代码（不凭记忆或推测）
- 制作对比清单（表格列/数据字段/配置项）
- 逐项验证一致性（数量/顺序/命名）

**表格重构**：列数、列名、列顺序必须与原始代码完全一致
**条件分支重构**：所有 if/else/switch 分支都必须处理，不能只处理一个分支就推测其他分支
**服务拆分重构**：先提取共享工具方法到独立类，再拆分子服务（防止循环依赖）

### 4. 提交物质量
- ❌ 提交调试残留（`console.log`, `print`, `fmt.Println`）
- ❌ 用“注释掉旧实现”代替删除代码
- ✅ 所有的 `TODO` 必须在任务结束前清理或转为正式 issue

### 5. 自动添加 AI 元数据
- ❌ 禁止添加 "Generated by Gemini" 或 AI 自动签名
- ❌ 禁止添加冗余的变更总结文档（除非明确要求）

---

## ✅ 架构与流程

### 1. Type-First (类型先行)
无论是编写新业务还是**修改现有逻辑**，必须遵循 **“修改数据结构 -> 修改逻辑 -> 修改调用/模板”** 的单向防呆流程：
- **定义先行**：编写前必须先定义（或更新）数据结构：
  - **Go**: struct / interface
  - **TypeScript**: interface / type
  - **Java**: DTO / VO / Entity
- **严禁越级调用**：禁止在未更新类型定义的情况下，凭直觉在前端模板或业务逻辑中调用不存在的字段（如 `user.nickname`），防范“幻觉属性”导致的编译失败。

### 2. 复杂任务分步确认
当任务涉及超过 3 个文件或修改核心逻辑时：
- 步骤 1：提出技术方案并说明影响
- 步骤 2：**等待确认** ⏸️
- 步骤 3：获得确认后分步实施

---

## 📂 决策原则

```
简单方案 > 复杂方案
复用现有 > 创建新的
直接实现 > 抽象封装
排查: 复现 -> 假设 -> 验证 -> 最小修复
```

**HTTP 错误诊断**：当第三方 API 返回 HTTP 错误且 body 为空、headers 极简（缺少目标 API 的标准 headers），优先怀疑 CDN/WAF/代理层拦截，而非 API 本身的业务错误。验证方法：同一 API 的 GET 请求是否正常（GET 正常 POST 异常 → HTTP 客户端兼容性问题）。
