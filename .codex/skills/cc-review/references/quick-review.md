# Quick Review

- 主要看当前 diff 和最可能出问题的相邻代码。
- 优先抓高风险点，不追求穷尽式扫描。
- 输出简短，但 findings 要明确。
- 默认使用简体中文标题和风险等级，不使用 `Findings`、`High`、`Medium` 这类英文标题作为主输出。

## 输出格式

```
## 改动概览
[文件列表和改动行数]

## 问题（必须修复）
- [ ] [文件:行号] 问题描述

## 建议（每条必须打标签，三选一）
- ✅ [文件:行号] 建议内容              ← 本轮立即修复
- 📝 [文件:行号] 建议内容 → 追踪：<去向>  ← 落地为技术债
- ❌ [文件:行号] 建议内容 → 理由：<理由>  ← 明确拒绝

## 结论
✅ 可以提交 / ⚠️ 建议修复后提交 / ❌ 需要修复
```

## 建议分类强制规范

避免反复 review 提同样建议：

| 标签 | 含义 | 必填字段 |
|------|------|---------|
| ✅ 本轮立即修复 | 本次 PR 内可低成本修复，纳入下一步 `$cc-fix` 入参 | 修复策略一句话 |
| 📝 落地为技术债 | 当前 PR 不修，必须有追踪去向 | `→ 追踪：<issue 编号 / .codex/tasks/tech-debt.md 条目 / next-iteration>` |
| ❌ 拒绝 | 明确不修 | `→ 理由：<原代码遗留 / 违反 [code-review] 约束 / 业务无关 / 风险大于收益>` |

**禁用措辞**（不允许悬空建议）：

- ❌ "留给下次迭代"、"后续可以考虑"、"建议未来"、"可在下一版处理"
- ❌ 只写建议不带去向

理由：相同建议在多轮 review 里反复出现会浪费 token 和注意力。所有建议都必须有明确归宿。


**quick 模式的 📝 项处置**：quick 模式不弹询问、不自动写入 `tech-debt.md`，所有 📝 项的「追踪去向」由用户手动追加到 `.codex/tasks/tech-debt.md`。如需自动归档，请改用 `$cc-review`（full 模式）。
