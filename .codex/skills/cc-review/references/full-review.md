# Full Review

- 除当前 diff 外，还要看受影响调用链和关键边界条件。
- 检查输入输出契约、异常路径和回归面。
- 总结可以简短，但不能替代 findings。

## 输出格式

```markdown
### 代码审查报告

**总体评估**：[简要说明变更的整体质量和主要发现]

### 问题发现

#### [Critical] 严重问题（默认全部本轮修复）
- **文件:行号**：[问题描述和原因]

#### [Improvement] 建议改进（每条必须打标签）
- ✅ **文件:行号** [建议]                    ← 本轮立即修复
- 📝 **文件:行号** [建议] → 追踪：<去向>      ← 落地为技术债
- ❌ **文件:行号** [建议] → 理由：<理由>      ← 明确拒绝

#### [Nit] 细节建议
- Nit: **文件:行号**：[小建议]（不纳入自动修复也不归档）

### 结论

[通过 / 需修改后通过 / 需重新设计]
```

## 处置询问

完成审查后，如结论非"通过"，按 4 选项询问用户：

```
1. 只修复严重问题（推荐）
   修复 [Critical]；[Improvement] 全部落到 .codex/tasks/tech-debt.md

2. 修复全部问题
   修复 [Critical] + [Improvement]

3. 全部归档为技术债
   不本轮修复，全部写入 .codex/tasks/tech-debt.md

4. 跳过
   仅保留报告
```

选择 1 或 3 时，**必须**追加 `.codex/tasks/tech-debt.md`（无则创建）：

```markdown
## YYYY-MM-DD review 归档

- [ ] [文件:行号] 问题描述（来源：commit <短 hash> 的 review）
```

## 建议分类强制规范

避免反复 review 提同样建议：

| 标签 | 含义 | 必填字段 |
|------|------|---------|
| ✅ 本轮立即修复 | 纳入 `$cc-fix` 入参 | 修复策略一句话 |
| 📝 落地为技术债 | 必须有追踪去向 | `→ 追踪：<issue / tech-debt.md / next-iteration>` |
| ❌ 拒绝 | 明确不修 | `→ 理由：<...>` |

**禁用措辞**：「留给下次迭代」「后续可以考虑」「建议未来」等悬空话术。
